PRACTICAL ONE MATLAB Simulink Integration

Aim

To successfully load, configure, and evaluate an external Fuzzy Inference System (.fis) block within a MATLAB Simulink simulation loop architecture.

Objective

To successfully integrate an offline-designed `.fis` variable definition model directly into a dynamic Simulink tracking canvas block parameter structure without syntax mismatch warnings.

Methodology

The workspace environment was initialized by invoking the fuzzy design application panel via the command line fuzzy. Inputs, membership shapes, and conditional rule arrays were configured before exporting the structure directly to the workspace under a dedicated variable name. Simulink was then opened by running simulink. A structural logic model was created using a Step Input block, a Scope visualization layout, and the native Fuzzy Logic Controller block dragged from the Fuzzy Logic Toolbox library. Double-clicking the controller block opened its parameter configurations, where the exact global variable string matching the exported workspace FIS structure was specified.

Observations
  1. The Fuzzy Logic Controller block automatically adjusts its input and output port counts instantly upon entering a valid FIS workspace structure key name.
  2. If the referenced `.fis` variable is cleared from the MATLAB active base workspace memory structure, the Simulink simulation execution environment crashes with an unresolved identifier exception code.
  3. The real-time runtime scope traces present smooth, non-linear system state transitions instead of sharp rectangular step adjustments, validating proper interpolation between membership rules.
Conclusion

Importing `.fis` structures into Simulink streamlines the transition from descriptive linguistic rule arrays to dynamic, closed-loop time simulations, simplifying verification processes for intricate engineering problems.

PRACTICAL TWO Toolbox vs M-File

Aim

To design, validate, and compare a functional Fuzzy Inference System implemented through both the interactive MATLAB GUI Toolbox and programmatic text scripts (.m file).

Objective

To verify mathematical parity and structural consistency between code-generated fuzzy systems and graphical user interface models by comparing their generated output surfaces.

Methodology

First, the interactive GUI application workspace was loaded via the terminal keyword entry fuzzy. Inputs, membership equations, and intersection rule matrices were applied using point-and-click operations. Second, an identical architectural layout was scripted programmatically in a standard text execution container (.m file) using native programmatic calls such as mamfis(), addInput(), and addRule(). Output validation loops were confirmed via the cross-surface analysis command gensurf().

Observations
  1. The Graphical User Interface Toolbox offers immediate, interactive validation of membership curves but lacks automated batch replication across varying ranges.
  2. The M-file programmatic environment loads complex rule blocks significantly faster than human input speeds and scales effortlessly across variable dimensions.
  3. Both construction methodologies generate identical continuous numerical value trends when evaluated against identical runtime data array sequences.
Conclusion

The GUI Toolbox serves as an excellent prototyping environment for initial design layouts, while script-driven generation is much better suited for high-throughput optimizations and embedding logic within algorithmic loops.

PRACTICAL THREE MATLAB Matrix Operations

Aim

To execute matrix calculations inside the MATLAB Command Line environment, emphasizing the functional differences between linear algebraic operations and element-wise array math.

Objective

To compute the exact matrix addition sum ($A+B$) and element-wise matrix multiplication array product ($A.*B$) for the specified system matrices $A$ and $B$.

Methodology

The exact numerical grid configurations were entered directly into the terminal stream prompt window environment:

A = [1 2 3; 4 5 6; 7 8 9];
B = [4 5 6; 7 8 9; 10 11 12];
result_addition = A + B;
result_element_multiplication = A .* B;
Observations
  1. The basic matrix addition call (A + B) processes calculations directly across individual coordinates, requiring identical row-by-column structures to avoid array dimension errors.
  2. Executing standard multiplication (A * B) uses row-by-column cross-product rules, whereas adding the dot-operator prefix (A .* B) bypasses vector algebra to scale items independently at matched positions.
  3. Attempting element-wise multiplication on mismatched dimension arrays causes an immediate runtime exception, whereas a regular multiplication call might succeed if inner matrix dimensions match ($M \times N \cdot N \times P$).
Conclusion

Understanding the distinction between standard matrix algebra and element-wise dot-notated modifiers is crucial for ensuring accuracy when mapping multidimensional transformations or tensor processing routines.

PRACTICAL FOUR Python Linear Regression

Aim

To design, optimize, fit, and validate a supervised Linear Regression Machine Learning model using Python's scikit-learn library tools.

Objective

To establish a supervised pipeline that calculates predictive intercept coefficients and minimizes mean squared variance errors across numerical point paths.

Methodology

A Python execution environment was set up by importing numpy, matplotlib.pyplot, and LinearRegression from sklearn.linear_model. The dataset vectors were defined, and the structural features were reshaped into a multi-row dimensional layout. The model parameters were optimized by running the .fit(X, y) learning routine. Predictions were generated across the target range, and a visual chart was rendered using the plt.scatter() and plt.plot() functions to display coordinate metrics.

import numpy as np
import matplotlib.pyplot as plt
from sklearn.linear_model import LinearRegression

X = np.array([1, 2, 3, 4, 5]).reshape(-1, 1)
y = np.array([2, 3.9, 6.1, 8.0, 10.2])

reg_model = LinearRegression().fit(X, y)
y_pred = reg_model.predict(X)
Observations
  1. The fit() optimizer automatically extracts the target slope intercept and directional weights matrix, isolating standard coefficients matching systemic deviations.
  2. The performance variance evaluation loop records an $R^2$ accuracy score approaching 1.0, proving a strong linear relationship between the input data columns.
  3. Plotting structural inputs directly against predicted arrays reveals a balanced line of best fit that evenly splits residual variances across all coordinate values.
Conclusion

Supervised linear regression models provide a highly reliable framework for predictive analytics when working with highly correlated continuous variables.
